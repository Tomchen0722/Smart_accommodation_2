# 資料分析特徵選擇說明（X 與 Y）

本文件說明「空房風險預測」模型為何選用這些欄位作為特徵 X 與目標 Y，
以及地理便利性與 NLP 分析各自的 X/Y，並誠實列出重要的分析限制。
（對應程式：`modules/ml_models.py`、`modules/geo_utils.py`、`modules/nlp_analysis.py`）

---

## 一、空房風險預測（監督式分類：Logistic Regression + Random Forest）

### 目標變數 Y ＝「是否高風險」（1 / 0）

Inside Airbnb 的公開資料**沒有直接的「這間房是否空置」欄位**，因此我們先用可觀測的
代理訊號合成一個 `risk_score`，再取最高一級（`risk_level == 高風險`，score > 0.60）作為
分類正樣本：

```
risk_score = 0.40 × (availability_365 / 365)        # 年度可訂天數越多 → 越可能空著
           + 0.30 × (1 − number_of_reviews / 100)   # 累積評論越少 → 越冷門
           + 0.20 × (1 − number_of_reviews_ltm / 20)# 近一年評論越少 → 近期越冷
           + 0.10 × (price / price_第95百分位)        # 定價相對越高 → 越難出租
risk_level = 低風險(<0.35) / 中風險(0.35–0.60) / 高風險(>0.60)
Y = (risk_level == 高風險) → 1，其餘 → 0
```

**為何這樣定義 Y：**
- 「空房風險」是本平台對房東最有價值的預測；缺乏真實出租結果時，用
  「可訂天數多、評論少、近期冷清、定價偏高」這四個公認與空置高度相關的訊號組成綜合分數，
  是常見的**弱標籤（weak label）／代理標籤**做法。
- 取「高風險」單一門檻做二元分類，讓模型能用標準分類指標（Recall/AUC 等）評估，
  也符合平台「找出最需要介入的房源」的營運目的。

### 特徵 X（共 11 維：7 個數值 + 4 個房型 one-hot）

| 特徵 | 選用理由 |
| --- | --- |
| `price` 每晚價格 | 定價高低直接影響需求與成交難易，是空房與否的核心變數。 |
| `minimum_nights` 最少入住晚數 | 門檻越高越限制客源，短租門檻與空置相關。 |
| `number_of_reviews` 評論總數 | 累積人氣與信任度的代理；越多通常越搶手。 |
| `reviews_per_month` 月均評論數 | 近期需求熱度／週轉速度，反映當前而非歷史的搶手程度。 |
| `calculated_host_listings_count` 房東房源數 | 區分專業／多房東與個人房東，經營強度不同、空置行為不同。 |
| `availability_365` 年度可訂天數 | 供給／空置最直接的訊號（可訂越多、實際被訂越少）。 |
| `number_of_reviews_ltm` 近12月評論數 | 近一年實際入住活躍度，比累積評論更貼近「現在」。 |
| `room_type`（one-hot：整棟／私人套房／共用套房／飯店客房） | 不同房型的需求結構與空置率差異大，需納入。 |

**為何選數值型 + 房型：** 這些欄位在資料中缺失率低、語意明確、且與「出租熱度／供給／
定價」直接相關，適合線性模型（LR，看係數方向）與樹模型（RF，看非線性與特徵重要度）。

### 未納入的欄位與原因
- **評分 `review_scores_*`**：缺失率高（無評論者為空），且與 `number_of_reviews` 高度相關，
  貿然納入會引入偏誤與大量缺值。
- **經緯度 `latitude/longitude`**：地理效果改由「便利性評分」另行處理（見第二節），
  避免模型用座標硬記特定地段而失去泛化性。
- **文字欄位（name/description/amenities 原文）**：屬非結構化資料，交由 NLP 或設施篩選處理。

### 前處理
- 房型以 `pd.get_dummies` 做 one-hot；LR 前用 `StandardScaler` 標準化（RF 不需要）。
- 類別不平衡以 `class_weight="balanced"` 校正。
- 8:2 分層抽樣切訓練／測試（後台可調整為三分：訓練/驗證/測試，另有 K-fold 交叉驗證）。

### ⚠️ 重要分析限制：標籤洩漏（label leakage）
`Y` 是由 `availability_365、number_of_reviews、number_of_reviews_ltm、price` 這四個欄位
合成的；而這四個欄位**同時也在特徵 X 裡**。因此模型很大程度是在「還原那條加權規則」，
這也是測試 AUC 高達約 0.997 的原因——**這個高分不代表真實世界的預測力**，而是特徵與
標籤定義重疊的結果。

正確解讀與可能的改進：
- 目前的 ML 層主要示範**完整的資料切割 → 訓練 → 評估流程**與可解釋性，而非宣稱能獨立預測空房。
- 若要做「真正的預測研究」，建議：(1) 取得**獨立的真實標籤**（如實際入住天數／訂房結果）；
  (2) 從 X 移除定義標籤用的欄位，只保留外生特徵（如 `minimum_nights`、`reviews_per_month`、
  `calculated_host_listings_count`、房型、便利性分數）；(3) 以時間切分（train 過去、test 未來）避免時序洩漏。
- 附帶一提：`reviews_per_month、minimum_nights、calculated_host_listings_count、room_type`
  這幾項**不在**標籤公式內，是目前提供獨立訊號的特徵。

## 二、地理便利性評分（規則式，無監督）
- **X**：房源經緯度 → 以 Haversine 計算到 7 類 PoI（捷運、公車、超商、餐廳、學校、診所、公園）
  的「最近距離」與「800m 內數量」。
- **輸出（可視為 Y）**：每類 0–10 的連續分數＝0.6×鄰近度＋0.4×密度；租客端再依勾選項目
  取平均得綜合便利分。採連續分數是為了讓不同房源可區分、使用者勾選能真正改變排序。

## 三、NLP 評論情感分析（VADER + jieba，無監督）
- **X**：`cleaned_comments`（評論文本）、`language_type`（語言別）。
- **輸出（可視為 Y）**：情感標籤（正面 compound≥0.05／中立／負面≤−0.05）、正負關鍵字 Top-N、
  平均情感分數；用於房源評論摘要與全站情感統計。

## 四、房東智慧建議（規則式決策，非機器學習）
- **X**：目前定價 vs 1KM 中位價、空房機率 P、周邊設施勾選率、季節。
- **輸出 Y**：三維度建議（價格優化／動態行銷／硬體升級），依 P 的門檻（<30/30–60/≥60%）觸發。
