# v2 模型特徵字典 — X / Y 定義、計算方式與取用理由

> 對應程式：`scripts/build_dataset_final.py`（特徵建構的唯一事實來源）
> 對應資料：`data/dataset_final.csv`（5,849 筆 × 50 欄）
> 本文件每一條公式皆與程式碼逐行對照，文件說的 = 程式算的。

---

## 一、目標變數 Y（模型要預測什麼）

### Y_vacancy — 未來一年空屋率（模型 A 迴歸目標）

```
Y_vacancy = availability_365 / 365，夾在 [0, 1]
```

- `availability_365`：Inside Airbnb 快照當下，該房源**未來 365 天**行事曆上仍可被預訂的天數。
- **取用理由**：可訂天數高 = 沒被訂走 = 滯銷風險的直接代理。這是「向前看」的變數（未來供給），符合預警平台的任務定義。
- **為何不用舊版 risk_level 當標籤**：v1 的 risk_level 由規則 risk_score 算出，而 risk_score 的 40% 權重正是 availability_365 —— 該欄同時又是特徵，模型等於在學自己的計分公式（標籤洩漏，指標虛高且無預測意義）。v2 把 availability_365 專職當標籤來源、徹底移出特徵。

### Y_high_risk — 高風險二元標籤（模型 B 分類目標）

```
Y_high_risk = 1 if Y_vacancy > 0.7 else 0        （全資料集正例占比 31.4%）
```

- **0.7 的由來**：產品規則「未來一年超過七成天數空著 → 觸發滯銷通知」。分類器輸出經 isotonic 校準後，再用驗證集在 Recall ≥ 0.80 約束下搜尋通知門檻（寧可多抓、不可漏抓）。

### 資料協定（哪些列進得了訓練）

| 步驟 | 筆數 | 規則與理由 |
|---|---|---|
| 原始 listings_cleaned | 6,241 | Inside Airbnb 台北快照（快照基準日 2025-09-30） |
| 剔除座標缺值 | −0 | 地理特徵必要條件 |
| 剔除目標缺值 | −0 | Y 無法計算 |
| **剔除經營未滿 1 年** | **−392 → 5,849** | `host_tenure_days < 365`。剛上架房源行事曆天生大開，availability 高不代表滯銷，屬占位偏差，混入會汙染標籤語義 |

### 已知限制（誠實揭露）

availability_365 是「快照時點的未來可訂供給」，包含房東主動封鎖行事曆的天數（自用、維修），與「賣不掉」不完全等價 —— 這是資料源的結構性噪音，也是誠實 R² 存在天花板的原因之一。

## 二、完整模型 X — 36 特徵（老房東用）

缺值策略：**保留 NaN 不補值**。HistGradientBoosting 原生支援缺值（把「缺」當一種訊號學習）；線性基準模型的中位數補值器放在訓練管線內、只 fit 訓練集，避免補值統計量洩漏測試資訊。

### 群組 1｜結構化特徵（13 個）— 房源的「硬體身分證」

| 特徵 | 繁中 | 計算方式 | 取用理由（預測假設） |
|---|---|---|---|
| accommodates | 可住人數 | 原欄位 | 容量決定客群（背包客/家庭/團體），不同客群滯銷風險結構不同 |
| bedrooms | 臥室數 | 原欄位（缺值保留 NaN） | 同上，房間規模 |
| beds | 床位數 | 原欄位 | 同上 |
| bathrooms_count | 衛浴數 | 由 bathrooms_text 解析（清洗階段已產出） | 舒適度硬指標 |
| is_shared_bath | 共用衛浴 | bathrooms_text 含 shared → 1 | 共用衛浴顯著壓低訂房意願，尤其非背包客群 |
| price | 每晚價格 | `"$1,234.00"` 去除 `$`、`,` → float | 定價是房東最可控的滯銷槓桿；模型需要絕對價格與（見競爭組）相對價格並存 |
| minimum_nights | 最短入住晚數 | 原欄位 | 門檻太高會篩掉短住需求 → 供給閒置 |
| maximum_nights | 最長可住晚數 | 原欄位 | SHAP 發現非線性：約 200 晚以下值越小風險越低（短租定位明確），超過 300 晚貢獻打平 —— 樹模型才吃得到這種型態 |
| min_nights_avg_ntm | 近期平均最短入住 | `minimum_nights_avg_ntm` 轉數值 | 未來一年的實際門檻設定（比靜態 minimum_nights 更貼近未來供給） |
| instant_bookable | 即時預訂 | `t/f → 1/0` | 免等房東確認 → 轉換率高 → 空屋率低的假設 |
| self_checkin | 自助入住 | amenities 字串（轉小寫）含 `self check-in / self-check-in / lockbox / keypad / smart lock` 任一 → 1 | 入住便利性；亦是「專業化經營」的訊號 |
| room_type_code | 房型 | 類別碼：`room_type` 依字母排序後編碼（排序固定保證可重現） | 整套/雅房/合住是完全不同市場 |
| neighbourhood_code | 行政區 | 類別碼：12 行政區依字母排序編碼 | 地段基本盤；樹模型可自行學習行政區間差異 |

### 群組 2｜評分特徵（7 個）— 口碑品質

| 特徵 | 繁中 | 計算方式 | 取用理由 |
|---|---|---|---|
| review_scores_rating | 總體評分 | 原欄位（1~5，缺值 NaN） | 口碑是訂房決策核心；「無評分」本身也是訊號（HistGB 直接學缺值） |
| review_scores_accuracy | 描述準確度 | 原欄位 | 照片/描述與實際落差 → 差評 → 滯銷的因果鏈 |
| review_scores_cleanliness | 清潔度評分 | 原欄位 | 旅宿差評第一大類 |
| review_scores_checkin | 入住體驗評分 | 原欄位 | 流程摩擦訊號 |
| review_scores_communication | 溝通評分 | 原欄位 | 房東服務品質 |
| review_scores_location | 地點評分 | 原欄位 | 「主觀感受的地段」，與客觀密度特徵互補 |
| review_scores_value | 性價比評分 | 原欄位 | 價格合理性的市場投票 |

### 群組 3｜競爭特徵（5 個）— 平台核心設計：「滯銷是相對的」

> 這組是專案的招牌：同樣條件的房源，在信義區和萬華區的命運不同 —— 風險必須放在**局部市場**裡衡量。

| 特徵 | 繁中 | 計算方式 | 取用理由 |
|---|---|---|---|
| price_pctl_nbhd | 同區同房型價格百分位 | 在（行政區 × 房型）群組內 `rank(pct=True)`，0~1 | 「你比同區同類貴幾成」比絕對價格更能解釋滯銷 |
| score_pctl_nbhd | 同區同房型評分百分位 | 同上，對 review_scores_rating 排名 | 相對口碑：4.5 分在高分區是後段班 |
| amenities_vs_median | 設施數/周邊中位數 | `amenities_count ÷ 同群組設施數中位數`（中位數為 0 → NaN 防除零） | 設施競爭力的相對值 |
| nbr_density_1km | 1km 房源密度 | BallTree（haversine 度量，座標轉弧度）查半徑 1000/6371000 弧度內鄰居數 **−1（扣掉自己）** | 局部供給強度：紅海或藍海 |
| nbr_density_same_type_1km | 1km 同房型密度 | 各房型**分別建樹**後同法計數 −1 | 真正的直接競爭者（整套只跟整套搶客） |

### 群組 4｜房東身分特徵（7 個）— 冷啟動模型移除的那一組

> 這組預測力最強（host_acceptance_rate 是全模型 SHAP 第一名），但**新房東全部空白** —— 這正是雙模型策略存在的原因，也是 GroupKFold 洩漏的來源（同房東多房源共享這組值）。

| 特徵 | 繁中 | 計算方式 | 取用理由 |
|---|---|---|---|
| host_acceptance_rate | 房東接受率 | `"95%"` 去 `%` ÷ 100 → 0~1 | 接受率低 = 挑客/懶得理 → 訂單流失；SHAP 全模型重要度 Top 1 |
| host_response_rate | 房東回覆率 | 同上 | 回不回訊息直接決定成交 |
| response_speed | 房東回覆速度 | 序數編碼：within an hour=4 / a few hours=3 / a day=2 / few days+=1 / 缺=0 | 比回覆率多一層「快慢」維度 |
| host_is_superhost | 超讚房東 | `t/f → 1/0` | 平台認證的綜合品質標章 |
| host_listings_count | 房東房源數 | 原欄位 | 職業房東 vs 個人房東的經營型態差異 |
| calculated_host_listings_count | 平台計算房源數 | 原欄位（平台端統計，較乾淨） | 同上；**也是新舊房東路由依據**：≤1 → 走冷啟動模型 |
| host_tenure_days | 房東經營天數 | `(快照基準日 2025-09-30 − host_since).days` | 經驗曲線；同時是資料協定的過濾欄位 |

### 群組 5｜經營用心度（4 個）— 「有沒有認真經營」的可量化代理

| 特徵 | 繁中 | 計算方式 | 取用理由 |
|---|---|---|---|
| desc_len | 房源描述字數 | `len(description)`，缺值視為空字串 → 0 | 描述空白/敷衍 → 資訊不足 → 轉換率低；字數是「用心程度」最廉價的代理 |
| host_about_len | 房東自介字數 | `len(host_about)` | 自介建立信任感 |
| neighborhood_overview_len | 周邊介紹字數 | `len(neighborhood_overview)` | 在地資訊豐富度 |
| amenities_count | 設施總數 | amenities（JSON 樣式字串）以逗號計數 +1；空清單 → 0 | 設施絕對量（相對量在競爭組） |

## 三、冷啟動模型 X — 35 特徵（新房東用）

```
35 = 36 − 房東身分 7 個 + 地點/房間 6 個
```

**為什麼要有第二個模型**：新房東沒有接受率、回覆率、超讚標章 —— 完整模型最強的一組特徵對他們全是缺值，預測會退化。冷啟動模型改用「地段條件 + 房間性價比」補位，這 6 個特徵是外部特徵實驗報告**實測驗證唯一有真實增益的組合**（GroupKFold R² +0.011、AUC +0.005 且變異極低 ±0.013~0.016）。

### 群組 6｜地點/房間特徵（6 個）

| 特徵 | 繁中 | 計算方式 | 取用理由 |
|---|---|---|---|
| hotel_count_1km | 1km 飯店數 | BallTree 對 680 筆 OSM 旅宿 POI（hotel/hostel/guest_house/motel）查半徑 1km 計數 | **反直覺發現**：原假設「飯店密→競爭兇→空屋高」被實測推翻（相關 −0.131）。飯店紮堆處是觀光精華地段 —— 飯店密度是「需求熱度代理」而非「競爭懲罰」 |
| hotel_count_500m | 500m 飯店數 | 同上，半徑 500m | 更細粒度的同一訊號（相關 −0.141，地點組最強） |
| airbnb_hotel_supply_ratio | 短租/飯店供給比 | `nbr_density_1km ÷ (hotel_count_1km + 1)`（+1 防除零） | 檢驗「短租供給相對飯店越高越滯銷」假設（實測弱訊號 +0.028，保留供模型自行取捨） |
| price_per_person | 每人單價 | `price ÷ accommodates`（0 → NaN） | 房間級性價比：跨容量可比較的價格 |
| price_per_bedroom | 每房單價 | `price ÷ bedrooms`（0 → NaN） | 同上；地點/房間版特徵重要度第 13 名，是這組最有用的 |
| beds_per_person | 每人床位數 | `beds ÷ accommodates` | 擁擠度/舒適度代理 |

> 完整模型**不加**這 6 個：實驗證明對老房東模型無統計顯著提升（訊號被 nbr_density、price_pctl 等既有競爭特徵吸收，屬冗餘）—— 符合 Occam，不為灌數字硬加特徵。

## 四、明確排除的欄位與理由（面試必考）

| 欄位 | 排除理由 |
|---|---|
| availability_30/60/90/365 | **目標洩漏**：Y 由 availability_365 算出，任何 availability 欄進特徵都等於偷看答案 |
| estimated_occupancy_l365d、estimated_revenue_l365d | 由 availability 直接推導，同屬洩漏 |
| number_of_reviews 系列、reviews_per_month | **反向因果疑慮**：熱門（低空屋）→ 評論多，評論數是結果不是原因；且與 host_tenure_days 高度共線。口碑「品質」面已由 7 個評分欄涵蓋 |
| risk_score、risk_level | v1 規則產物，即為本次修正的洩漏標籤 |
| photo_design_sense（CLIP 影像美感） | 重跑需下載 5,849 張圖（400+ 分鐘、CDN 限速），且原報告顯示其增益有限；本版為 36 特徵並全程誠實標註 |
| license、host_verifications、URL/名稱類 | 行政/識別欄位，無預測假設 |

## 五、一句話總結（可直接放簡報）

> Y 是「未來一年空屋率」這個向前看的真實目標；X 的設計哲學是**四層假設**：房源硬體是什麼（結構化）、市場怎麼評價它（評分）、它在局部市場的相對位置（競爭）、誰在經營與多用心（房東+用心度）—— 新房東缺最後一層，就用「地段需求代理 + 房間性價比」補位。每個特徵都有明確的預測假設、可重現的計算公式，以及被 GroupKFold 誠實協定與 SHAP 驗證過的去留理由。
